# BlackPanda CRM (React + Node + PostgreSQL)

Полный переезд с версии на HTML+Firebase на архитектуру StitchFlow: React/Vite/Tailwind
на фронте, Express + PostgreSQL на бэке.

## Важно
Этот код не был запущен и протестирован автоматически (в среде подготовки не было
доступа в интернет для `npm install`/подключения к БД). Перед боевым использованием
пройдите шаги ниже и сообщите, если что-то не заведётся с первого раза — пришлите
текст ошибки, поправим.

## Локальный запуск

1. Поднимите PostgreSQL локально (или используйте Railway/Render/Supabase).
2. `cp backend/.env.example backend/.env` и впишите свои данные подключения + `JWT_SECRET`.
3. Установка зависимостей:
   ```
   npm run install:all
   ```
4. Применить схему БД:
   ```
   npm run migrate
   ```
5. Создать первого пользователя (админа):
   ```
   npm run seed
   ```
   По умолчанию: `admin@blackpanda.local` / `changeme123` — обязательно смените пароль
   после первого входа, либо задайте `SEED_EMAIL`/`SEED_PASSWORD`/`SEED_NAME` перед запуском.
6. Запуск в двух терминалах:
   ```
   npm run dev:backend    # http://localhost:4000
   npm run dev:frontend   # http://localhost:5173 (проксирует /api на бэкенд)
   ```

## Деплой (например, на Railway)
`railway.toml` уже настроен: он собирает фронтенд, ставит зависимости бэкенда,
прогоняет миграцию и запускает сервер, который сам раздаёт собранный фронт
(`frontend/dist`) — то есть один сервис вместо двух.

Не забудьте добавить сервис PostgreSQL и переменные окружения: `DATABASE_URL`
(Railway подставит сам, если добавить Postgres-плагин), `JWT_SECRET`.

## Новое (по вашему запросу)
- **Роли и права доступа (как в DLE):** admin / staff (сотрудник) / accountant (бухгалтер) / client (клиент).
  Админ в разделе «Админ-панель → Права доступа» настраивает для каждой роли и для каждого
  раздела (склад, клиенты, предзаказы, продажи, касса, настройки) — можно ли его видеть и
  можно ли в нём редактировать. Плюс точечные права на конкретного человека, которые
  перекрывают права его роли (например, один сотрудник — без доступа к кассе, хотя все
  остальные сотрудники видят её).
- **Админ-панель:** создание пользователей с паролем прямо в интерфейсе (без Firebase Console),
  включение/отключение доступа, смена роли и пароля.
- **Роль «клиент»:** у клиента, которому вы создали логин, есть свой урезанный портал (`/portal`)
  — видит только свои предзаказы, покупки и долг, ничего больше.
- **Двуязычность RU/ZH** — переключатель в сайдбаре и на экране входа, сохраняется в браузере.
- **Рассчитано на 10 одновременных пользователей** — пул соединений с БД увеличен (`max: 20`),
  все операции передачи товара идут в транзакциях с блокировкой строки (`FOR UPDATE`), поэтому
  два человека не смогут случайно продать один и тот же серийник дважды.
- **Дизайн под ваш референс:** красно-чёрная палитра, логотип-иллюстрация панды
  (`frontend/public/panda-logo-icon.png` — маленькая иконка, `panda-logo-full.png` — большая
  иллюстрация внизу сайдбара), адаптивная вёрстка сайдбара (иллюстрация масштабируется через
  `aspect-ratio`, а не обрезается фиксированной высотой, как в статичном макете).
- **Реальный график продаж/прибыли по месяцам** и **топ-5 моделей по продажам** на дашборде —
  не бутафория из макета, а настоящие агрегирующие запросы к БД (`GET /api/dashboard`),
  отрисованные через `recharts`.

## Новые разделы (полноценный функционал, не заглушки)
- **Поставщики** — карточки поставщиков (контакт, WeChat, страна), счётчик поставленных единиц.
  Серийники теперь можно привязать к поставщику (`serials.supplier_id`).
- **Сотрудники** — HR-карточки (должность, телефон, дата найма, зарплата), опционально привязаны
  к учётной записи в системе (`users`). Это отдельная сущность от логинов в Админ-панели —
  не у каждого сотрудника есть доступ к CRM.
- **Финансы** — P&L (выручка / себестоимость / валовая прибыль / расходы / чистая прибыль),
  график по месяцам, разбивка расходов по категориям. Расходы в Кассе теперь можно
  категоризировать (`cash_log.category`) — это и питает разбивку в Финансах.
- **Аналитика** — распределение продаж по брендам (донат-диаграмма), топ-8 клиентов по обороту,
  средний чек, топ моделей по продажам.
- **Отчёты** — отчёт по продажам за период + экспорт в CSV, отчёт по остаткам склада.
- **Импорт** — вставка CSV (через `papaparse`) для массового добавления клиентов и моделей склада,
  с предпросмотром перед импортом.
- **Журнал действий** — аудит ключевых операций (продажа, передача по предзаказу, движение кассы,
  создание клиента/модели/пользователя) — кто, что и когда сделал.

Все новые разделы полностью встроены в систему прав доступа: админ в Админ-панели решает,
какая роль (или конкретный человек) видит и может редактировать каждый из них — точно так же,
как уже работающие Склад/Клиенты/Кассa.

## Функционал, перенесённый из большой HTML-версии (по вашему запросу — всё по списку)

**1. Сканер + визард продажи**
- Раздел «Сканер»: 3 шага — скан серийников (камера через `@zxing/library` или ручной ввод) → выбор клиента → итог с ценами по каждой модели и способом оплаты
- Резервирование товара за клиентом с дедлайном (видно на странице Склада, автоснятие резерва по истечении срока при следующем обращении к API)
- Печать чека после продажи и QR-этикеток серийников (кнопка 🏷️ на карточке модели)

**2. Долги / баланс клиентов / банковские счета / сплит-оплата**
- В карточке клиента: баланс предоплаты (пополнение/списание/обнуление с историей), долги с датой погашения, кнопка «Погасить долг», кнопка «Напомнить» (шлёт в Telegram)
- Банковские счета (Сбер/Альфа/Т-банк) на странице Кассы — ручная корректировка, автоматическое пополнение при продаже
- При продаже — любой способ оплаты: полностью (нал/один из банков), частично (остаток в долг с датой), сплит (часть нал + часть на карту), с баланса клиента

**3. Курс ЦБ РФ + графики аналитики + рассылка + QR/печать**
- На странице Аналитика: график продаж/себестоимости/маржи с переключением периода (7д/30д/1г/по годам), график своего курса против курса ЦБ РФ (кнопка «Обновить курс ЦБ» тянет данные с cbr-xml-daily.ru прямо из браузера)
- Раздел «Рассылка»: отправка сообщения всем клиентам с Telegram (или части), шаблоны сообщений, авто-генерация текста со списком товаров в наличии
- Печать чеков и QR-этикеток

## Сортировка и перетаскивание (drag-and-drop)
- **Справочник**: бренды, серии и все плоские списки (CPU/GPU/RAM и т.д.) по умолчанию идут по алфавиту;
  новый элемент автоматически встаёт в алфавитную позицию относительно соседей. Любой элемент можно
  перетащить мышкой или пальцем на телефоне (иконка ⠿), новый порядок сохраняется на сервере.
- **Порядок меню**: Админ-панель → вкладка «Порядок меню» — админ перетаскивает разделы в нужном порядке,
  это применяется сразу ко всем пользователям.
- Оба перетаскивания реализованы на pointer events (не HTML5 drag-and-drop), поэтому одинаково
  работают и мышкой на компьютере, и пальцем на телефоне/планшете.
- Разделы, к которым у пользователя нет прав на просмотр, не просто скрыты — они вообще не попадают
  в список для рендера (проверка `can(page, 'view')` в Layout.jsx), то есть их не видно в меню совсем.

## Звук при сканировании + карточка серийника с историей
- В «Сканере»: успешное нахождение серийника — короткий высокий сигнал, ошибка/не найден — низкий сигнал (Web Audio API, без файлов)
- Каждый отсканированный серийник кликабелен — открывает карточку серийника в новой вкладке (не мешает продолжить оформление продажи в текущей)
- Карточка серийника (`/serials/:id`): полные характеристики, статус, гарантия, дата поступления/продажи,
  заметки, переключатель уведомлений гарантии, кнопка «Возврат» (если продан — фиксирует от какого клиента и по какой причине)
- Полная история изменений статуса серийника — кто, когда, что изменил, включая продажи с именем клиента и возвраты

## Импорт из старой HTML/Firebase-версии
В разделе «Импорт» появился блок «Импорт из старой версии»:
1. В старом приложении нажми «Бэкап данных» — скачается JSON-файл со всем `S` состоянием
2. Загрузи этот файл в новый блок на странице Импорт
3. Перенесутся: клиенты (с балансом), модели ноутбуков и фото, серийники с историей статусов,
   продажи с позициями, долги клиентов, касса, курс и его история, банковские счета
4. Серийники защищены от дублей уникальным номером — но клиенты и продажи не защищены,
   поэтому **запускать импорт нужно только один раз**

Что не переносится (можно добавить отдельно, если нужно): резервы, шаблоны сообщений,
журнал действий старой версии, сервис/ремонт, "старение склада".

## Статусы товара — теперь управляемый список (Справочник)
Вместо жёстко зашитых 4 статусов — полноценный список в Справочнике, с настройкой,
к какой "корзине" относится каждый статус для агрегатов на складе/дашборде:
- **В наличии** (instock) — считается как остаток, можно продать. Изначально сюда входят
  "На складе", "Склад (восст.)", "Новый (Коробка повр.)"
- **В пути** (intransit), **Резерв** (reserved), **Продано** (sold)
- **Не считать в остатках** (other) — просто информационные статусы: "Возврат", "Гарантия КНР",
  "На ремонте", "Потерян"

Логика продажи/возврата: при продаже в истории серийника фиксируется, какому клиенту продано.
При возврате сотрудник указывает причину и **сам выбирает**, какой статус присвоить дальше
(например "Склад (восст.)", "Гарантия КНР" или любой другой) — статус не откатывается
автоматически на "На складе".

Можно добавлять свои статусы (Справочник → Статусы товара), выбирая для каждого, входит ли он
в остаток склада или нет, и перетаскивать их в нужном порядке.

## Двуязычный Справочник (RU/中文)
Раздел «Переводы» из старой версии не переносился — там просто хранился перевод статичного
интерфейса, а у нас это уже сделано автоматически (i18n на весь UI). Вместо этого сделали то,
что реально требует ручного ввода: **бренды, серии, характеристики (CPU/GPU/RAM/накопитель/
цвет/экран) и статусы товара теперь двуязычные** — рядом с каждым значением есть поле «中文»,
куда вписывается перевод. При переключении интерфейса на китайский везде, где это значение
показывается (Склад, карточка модели, карточка серийника), вместо русского текста подставляется
китайский перевод — если он заполнен, иначе остаётся русский текст как запасной вариант.

Стартовые данные (10 брендов, базовые CPU/GPU/RAM/накопители/цвета/экраны и все статусы товара)
уже пришли с переводами — их можно поправить прямо в списке, если где-то неточно.

## Исправление: передача по предзаказу — та же логика оплаты, что в Сканере
Раньше страница передачи товара по предзаказу была на старой упрощённой логике (только 4 способа
оплаты без выбора счёта, без сплита, а долг при частичной оплате писался в поле, которое больше
нигде не отображается — реальный баг). Теперь:
- Способы оплаты полностью совпадают со Сканером: полностью (нал/любой счёт), частично (в долг
  с датой — теперь корректно попадает в таблицу `debts`, видно в Финансах/карточке клиента),
  сплит (нал+перевод с выбором счёта), с баланса клиента
- Звук при сканировании серийника (нашёл/ошибка), клик по серийнику открывает его карточку/историю
  в новой вкладке, печать чека после подтверждения передачи
- Снятие резерва при передаче, если серийник был зарезервирован

## Исправления по интерфейсу (важный технический баг)
- **Настоящая причина «маленького окошка» в Справочнике**: кастомные классы (`.inp`, `.card`, `.btn`
  и т.д.) были объявлены в `index.css` обычным CSS уже ПОСЛЕ директив `@tailwind` — из-за этого
  `w-full` внутри `.inp` перебивал `flex-1`/`w-24` и подобные утилиты Tailwind везде, где они
  использовались вместе. Обернул кастомные классы в `@layer components` — теперь утилиты Tailwind
  всегда имеют приоритет, как и задумано изначально.
- Уменьшены отступы: карточки (`p-5`→`p-4`), кнопки/поля ввода компактнее, пункты бокового меню
  (`py-3`→`py-2`), сам сайдбар и шапка меньше «дышат» воздухом. Общий `line-height` текста слегка
  уменьшен (1.4). Добавлен класс `.inp-sm` (использовался по всему проекту, но нигде не был объявлен).

## Светлая/тёмная тема
Кнопка ☀️/🌙 рядом с переключателем языка в сайдбаре. Технически: вся цветовая палитра
переведена на CSS-переменные (`index.css`, `tailwind.config.js`) — переключение темы меняет
только значения переменных на `<html>` (класс `.light`), поэтому это работает во **всей**
системе сразу, без правки каждой страницы. Также приведены к переменным цвета графиков
(Recharts) на Дашборде, Финансах, Аналитике и Истории клиента — сетка/оси/тултипы теперь тоже
меняются вместе с темой. Выбор темы сохраняется в браузере и применяется мгновенно при
следующей загрузке (без "вспышки" неправильной темы).

- **Дашборд** — график курса юаня (раньше было только 7д/30д/1г)
- **История клиента** — было Сегодня/7дней/30дней/90дней/Год, теперь единообразно + добавлено
  «Всё время» (с начала работы с этим клиентом)

## Предзаказы — переработана логика ценообразования (юаневый долг)
- **Цена позиции считается сама**: (себестоимость ¥ + логистика ¥200 или ¥300) × (1 + наценка%).
  Наценка зависит от того, сколько клиент вносит предоплаты при заказе:
  0% предоплаты → 9% наценка, 50% → 6%, 100% → 3% — чем меньше платит сейчас, тем выше наценка
- В самом предзаказе видна разбивка по каждой позиции: `¥{себестоимость} + логистика ¥{200/300}
  + {наценка}% = ¥{итог} ≈ {рубли}`, и общая сумма по всему заказу
- **Остаток долга хранится и фиксируется В ЮАНЯХ, а не в рублях.** Если товар стоит ¥1000 и клиент
  заплатил ¥500 по курсу 12, а курс назавтра стал 13 — система при следующей оплате остатка сама
  пересчитает нужную сумму в рублях по **текущему** курсу (500 × 13, а не по старому курсу заказа).
  Это касается и точечного погашения долга в карточке клиента, и погашения "всех долгов разом",
  и списков должников на Дашборде/Финансах/Статистике — везде юаневые долги теперь пересчитываются
  по актуальному курсу, а не застревают на курсе создания
- Оплату остатка можно вносить в любой момент (кнопка «Внести оплату» в карточке предзаказа —
  частично или полностью) или прямо при передаче товара (та же логика оплаты, что в Сканере:
  нал/банк/сплит/долг/баланс — просто применяется теперь к оставшейся сумме по курсу на день
  передачи)
- Добавлена явная кнопка отмены предзаказа

## Уточнения по валютам
- **Статистика**: суммы теперь всегда показываются вместе — «¥X ≈ Y ₽» (не выбор одной валюты
  по языку интерфейса, а обе сразу, как в предзаказах)
- **Долг клиента**: у одного клиента могут быть и рублёвые, и юаневые долги одновременно —
  в карточке клиента у каждого долга теперь явная подпись: «Долг в рублях» или «Долг в юанях ≈
  рублей». Юаневые долги (обычно из предзаказов) при погашении/редактировании пересчитываются
  по курсу на день оплаты — рублёвые остаются как есть, без пересчёта.

## История цены товара + мини-график

## Исправление импорта из старой версии — продажи без клиента
Продажа в новой системе обязательно привязана к клиенту (так было спроектировано изначально —
у нас нет понятия "анонимная продажа"). Если в старом бэкапе есть продажа без указанного клиента
или со ссылкой на уже удалённого клиента — импорт раньше падал с ошибкой
`null value in column "client_id"`.

Теперь: если у продажи нет валидного клиента, импорт автоматически создаёт (один раз за весь
импорт) служебного клиента **«Без клиента (импорт)»** и вешает такие продажи на него — вместо
падения. Все прочие данные из бэкапа продолжают импортироваться как обычно. После импорта такие
продажи будут видны в Продажах у этого служебного клиента — если нужно, их можно вручную
переназначить на правильного клиента.

Заодно поправил соседний баг в этом же блоке (импорт долгов клиента) — там была ссылка на
переменную, которой не существовало в этой области видимости, что могло уронить импорт при
долге, заданном в юанях без суммы в рублях.

## Защита от пустого/чёрного экрана при сбое
Не смог воспроизвести проблему с чёрным экраном на карточке товара напрямую (нет доступа к
твоему браузеру), поэтому сделал две вещи:

1. **Error Boundary на весь сайт** (`components/ErrorBoundary.jsx`) — теперь если где-либо
   в приложении произойдёт непредвиденный сбой рендера, вместо полностью пустого/чёрного экрана
   покажется понятное сообщение "🐼💥 Что-то пошло не так" **с текстом самой ошибки** и кнопкой
   "Обновить страницу". Если проблема повторится — просто пришли скриншот этого сообщения,
   в нём будет точный текст ошибки, и я смогу починить прицельно, а не гадать.
2. **Укрепил историю цены и мини-график** — добавил защиту от некорректных/неожиданных данных
   (не-числа, пустые значения), плюс карточка товара теперь показывает понятную ошибку вместо
   вечной "Загрузки...", если запрос к серверу не удался.

## Найдена настоящая причина — подсказки справочника и, вероятно, чёрный экран
В `Warehouse.jsx` хук справочника был подключён неполностью:
`const { tr } = useLibraryText();` — а сам справочник (`lib`) нигде не забирался.
При этом дальше в коде `lib` использовался в нескольких местах (построение подсказок бренда,
серии, CPU, RAM, GPU, накопителя, цвета, экрана) — обращение к необъявленной переменной.

Это не просто "подсказки не показываются" — это **гарантированный сбой при каждом открытии
страницы Склад**, потому что `lib` фигурировал даже в массиве зависимостей `useMemo`, а не
только внутри условий. Такая ошибка укладывается в то, что раньше выглядело как пустой/чёрный
экран.

Исправлено: `const { lib, tr } = useLibraryText();`. Проверил весь остальной проект на такую же
ошибку (неполную деструктуризацию хука справочника) — больше нигде такого нет, только в этом
одном месте.

## Полоса сверху (адресная строка) и нижняя навигация
По скриншотам разобрался в обоих запросах:

1. **Полоса сверху — это не наш сайт, а сама Safari.** Адресная строка браузера показывается,
   когда сайт открыт как обычная вкладка. В старой версии её не было, потому что сайт был
   добавлен на главный экран телефона и открывался оттуда как отдельное приложение — тогда
   iOS прячет всю браузерную обвязку. Теперь сайт настроен как полноценное PWA-приложение
   (`manifest.json`, мета-теги `apple-mobile-web-app-capable` и т.д.) — **чтобы адресная строка
   пропала, нужно открыть сайт в Safari → «Поделиться» → «На экран Домой» и дальше запускать
   именно с этого значка на рабочем столе**, а не из обычной вкладки браузера.
2. **Нижняя навигация на мобильном** — добавлена, как в старой версии: Главная / Сканер /
   Склад / Клиенты / Ещё (последняя открывает то же боковое меню со всеми остальными разделами).
   Учитывает вырез экрана и полосу-индикатор Home снизу (safe-area) — актуально именно
   в полноэкранном PWA-режиме.

## Объединение дублей моделей (после импорта из старой версии)
Если импорт создал две карточки для одной и той же реальной модели (например, старая система

## Опасная зона — очистка склада/системы (только для тебя)
Спрятал в **Админ-панель → вкладка «⚠️ Опасная зона»** (видна только роли admin, как и вся
Админ-панель). Требует отдельный секретный код, который знаешь только ты — задаёшь его один раз
в этой же вкладке, потом он нужен для любого разрушительного действия. Плюс на каждое действие —
дополнительное текстовое подтверждение (нужно набрать ровно «УДАЛИТЬ СКЛАД» или «УДАЛИТЬ ВСЁ»),
чтобы точно не нажать случайно.

Два метода:
1. **Очистить склад** — удаляет модели, серийники и связанные с ними продажи/предзаказы.
   Можно указать диапазон дат добавления модели — тогда удалятся только модели, добавленные
   именно в этот период (например, только то, что попало дублями при недавнем импорте),
   остальной склад не тронется. Без дат — сносит весь склад целиком.
2. **Полная очистка системы** — сносит вообще всё: клиентов, склад, продажи, предзаказы, кассу,
   долги, сервис, рассылки, журнал действий. Пользователи, права доступа и Справочник (бренды/
   переводы) остаются — чтобы систему можно было сразу продолжить использовать с чистого листа.

Код доступа хранится захэшированным (как пароли пользователей), не в открытом виде.

## Удаление конкретного импорта (точнее, чем «Опасная зона»)
Теперь при каждой загрузке файла бэкапа старой версии все созданные им записи (клиенты,
модели, серийники, продажи, долги, касса) помечаются одним общим ID партии. В разделе
**Импорт** под формой загрузки появляется **«📜 История импортов»** — список всех загрузок
с датой и количеством, и у каждой кнопка **«🗑️ Удалить эту партию»**.

Удаление затрагивает **только эту конкретную партию**, не трогая ничего добавленного вручную
до или после — в том числе:
- Клиенты из партии удаляются, только если с момента импорта у них не появилось реальных
  продаж/долгов/предзаказов (иначе останутся — чтобы не потерять настоящую активность)
- Модели, серийники, продажи, долги и записи кассы из этой партии удаляются полностью

**Важно:** это работает для импортов, сделанных **после** этого обновления — партии, уже
загруженные раньше (до появления этой функции), нельзя выборочно отследить и удалить именно
так, для них по-прежнему нужна «Опасная зона» → «Очистить склад» (по дате или целиком).

## Курс юаня: автоматическое обновление + сравнение с ЦБ РФ
Теперь понял мысль правильно — вернул график сравнения, как было в старой версии.

## Долги от обычных продаж теперь тоже в юанях + явное частичное погашение
Понял правильно (не «логи», а «логика») — долг от частичной оплаты **обычной продажи** (не
только предзаказа) теперь тоже фиксируется в юанях по курсу продажи, а при погашении позже
пересчитывается по курсу на день оплаты — ровно как ты описал сценарий: продал ноутбук, клиент
ничего не заплатил сразу (0), приходит через время и платит часть — сумма к оплате в рублях
всегда будет пересчитана по сегодняшнему курсу, а не застынет на курсе продажи.

**Частичное погашение было и раньше технически возможно, но пряталось** за мелкой ссылкой,
открывавшей системное окошко браузера (`prompt()`) — легко было не заметить. Переделал в явное
поле суммы прямо в карточке долга + две понятные кнопки: **«Внести часть»** (свою сумму) и
**«Погасить полностью»** — теперь видно сразу, без скрытых диалогов.

## Найдена и исправлена серьёзная ошибка — «пропавший» товар в остатках
По твоему скриншоту нашёл настоящую причину сразу двух связанных жалоб:

**Общий остаток на складе (Дашборд, Статистика) считался НАПРЯМУЮ из таблицы серийников, вообще

больше, чем должно быть», и «карточка пропала, не могу найти». Исправлено в трёх местах
(Дашборд — общий остаток и спецстатусы, Статистика — общий остаток).

**Архивные карточки теперь видны** — список Склада больше не скрывает удалённые модели,
а показывает их **внизу списка, с разделителем** «⬇️ Нет в наличии / архив» (там же — карточки
с нулевым остатком без удаления). У архивных карточек — пометка «Архив» и кнопка
**«↩️ Восстановить»**, если удалили по ошибке.

**Бренды в распределении** — не трогал, ты сказал, что уже поправил вручную.

## Реализованы 5 предложенных функций

### 1. Предупреждение о проблемном серийнике перед продажей
В Сканере при сканировании серийника, у которого была история обращений в сервис (хотя бы
одно, по твоему решению — "лучше перебдеть"), под строкой появляется жёлтое предупреждение:
«⚠️ У этого устройства было N обращение(ий) в сервис — стоит предупредить клиента заранее»,
с пометкой, если среди них был гарантийный случай.

### 2. «Отложенный интерес» клиента (вишлист без предоплаты)
На карточке товара — кнопка **«👀 Клиент ждёт эту модель»**: выбираешь клиента, необязательный
комментарий, сохраняешь. Никаких обязательств по оплате. Когда модель появится на складе —
клиенту (если есть Telegram) автоматически уходит уведомление, а тебе — задача. На карточке
клиента виден список того, что он ждёт, со статусом (ждём / уже в наличии / уведомлён).

### 3. Напоминание клиенту о скором окончании гарантии
Новый фоновый процесс на сервере (проверяется раз в сутки, без кнопок): за 2 недели до конца
гарантии клиенту с указанным Telegram автоматически уходит напоминание. Каждому серийнику —
только одно напоминание (флаг в базе), чтобы не спамить повторно.

### 4. Источник привлечения клиента
Новое поле при создании/редактировании клиента — «Откуда пришёл» (Avito / Сарафан / Telegram /
Маркетплейс / Другое). Показывается на карточке клиента рядом с городом.

### 5. Публичная ссылка «поделиться моделью»
На карточке товара — кнопка **«🔗 Ссылка для клиента»**, копирует в буфер короткую публичную
ссылку вида `сайт/p/ID`. По ней — отдельная страница **без входа в систему**: фото,
характеристики, цена в ₽, наличие. Никаких внутренних данных (себестоимость, серийники)
не показывается — только то, что безопасно видеть клиенту.

## КРИТИЧЕСКИ ВАЖНО: исправлена моя ошибка — склад показывал пусто

Это была моя вина — регрессия от вчерашней оптимизации запроса склада. Настоящая причина:
в запросе группировка (`GROUP BY`) включала колонку с типом `json` (результат `json_agg()` для
истории цены), а у типа `json` в PostgreSQL **в принципе нет оператора сравнения** — из-за этого
весь запрос падал с ошибкой на уровне базы данных, склад отдавал 500-ю ошибку, а фронтенд эту
ошибку никак не показывал — просто оставался пустым экраном «Нет ноутбуков». Один из тех
случаев, которые я не могу поймать без доступа к живому PostgreSQL (о чём предупреждал ранее).

**Исправлено полностью:**
- Запрос переписан так, что внешняя группировка теперь вообще не нужна — остатки, продажи за
  30 дней и история цены считаются каждый в своём отдельном блоке и просто присоединяются
  один-к-одному к каждой модели, без единого GROUP BY на пересечении разных типов данных
- Заодно добавил обработку ошибки на самой странице Склада — если что-то подобное всё же
  случится в будущем, вместо молчаливого «Нет ноутбуков» появится явное сообщение с текстом
  ошибки, и будет сразу понятно, что дело не в реально пустом складе

причину.

## Курс в шапке — настоящая причина найдена и исправлена

Нашёл реальную причину (не про кэширование — про это уже не было): курс в шапке (и на ПК,
и на мобильном) запрашивался с сервера **только один раз**, в момент первого входа в систему.
Шапка/меню — общий каркас, который не пересоздаётся при переходе между разделами (обновляется
только содержимое страницы) — поэтому этот единственный запрос никогда не повторялся, сколько
бы вкладок ни переключал.

**Исправлено** (вариант 1, как ты выбрал): курс теперь запрашивается заново **при каждом
переходе на другую страницу** — и на ПК (`Header.jsx`), и на мобильном (`Layout.jsx`).
Перешёл в другой раздел — курс в шапке уже актуальный, без необходимости обновлять страницу
руками.

## Исправлена настоящая причина «долг не виден в Финансах»

Нашёл обе причины по присланным скриншотам:

### 1. Как появился отрицательный долг
«Редактировать долг» открывала окошко «Новая сумма долга», по умолчанию подставляя **полную**
сумму долга (не остаток). Если по ошибке ввести туда «сколько осталось доплатить» вместо
полной суммы — новая сумма становится меньше уже оплаченного, и остаток уходит в минус.
**Исправлено:** бэкенд теперь не даёт установить сумму долга меньше уже оплаченной — вернёт
понятную ошибку. Подсказка в окошке ввода тоже уточнена — явно показывает, сколько уже оплачено.

### 2. Почему клиент с реальным долгом пропадал из Финансов и Дашборда
Списки должников считали сумму **всех** долгов клиента вместе, и только потом проверяли,
больше ли она нуля. Если у клиента был один отрицательный (сломанный) долг и один нормальный —
в сумме могло получиться отрицательное число, и клиент **пропадал из списка целиком**, хотя
реальный долг был. **Исправлено** — теперь каждый долг вносит в общую сумму не меньше нуля
(отрицательный долг просто не учитывается, а не вычитается из остальных) — обе Финансы и
Дашборд.

### Заодно защитил карточку клиента
Долг с уже отрицательным/нулевым остатком (в том числе тот самый, что был показан на скриншоте)
теперь не отображается как активный долг с кнопками оплаты — как только это исправление
попадёт на сервер, у этого конкретного клиента карточка сама покажет только реальный
непогашенный долг (¥89 128), а сломанная запись перестанет мешать.

## Исправлены оба пробела в способах оплаты (Сканер)

### «С баланса» — остаток теперь честный долг
Если баланса клиента не хватает на всю сумму — остаток теперь **уходит в долг** (в юанях, как
и остальные долги), а не тихо считается «как будто уже оплачено наличными». Плюс в интерфейсе
при выборе этого способа теперь сразу видно: «С баланса спишется: X ₽» и, если не хватает,
«Баланса не хватает, в долг уйдёт: Y ₽» — до подтверждения продажи, а не после.

### «Разделённая» (наличные + перевод) — теперь проверяется, что сумма сходится
- На бэкенде — если наличные + перевод не равны итоговой цене продажи, продажа не проходит,
  возвращается понятная ошибка с точными цифрами
- На фронтенде — живой счётчик «Введено: X из Y ₽», зелёным если сходится, красным с пометкой
  «не сходится!» если нет — и кнопка подтверждения заблокирована, пока суммы не совпадут

## График цены убран из списка на Складе, остался в карточке товара

В таблице на Складе рядом с ценой больше нет мини-графика — освободилось место. В самой
карточке товара (при клике на модель) график остался на месте, в двух видах: рядом с ценой
продажи и отдельно в блоке «История цены».

## Предзаказ — не видно характеристик при выборе модели

Та же самая проблема, что уже чинили для «Прихода товара» — при выборе модели для предзаказа
в списке было только «Бренд Серия», без CPU/RAM/накопителя/видеокарты/цвета/сенсора. Если у
одной серии несколько вариантов — было невозможно понять, какой именно выбираешь.
Исправлено — теперь в выпадающем списке видно всё это рядом с названием.

## Финансы — «Погашение долга» теперь показывает только должников

Нашёл причину — селектор клиента в блоке «Операция» был один общий на все три действия
(пополнить баланс / добавить долг / погасить долг), и всегда показывал полный список всех
клиентов. Логично для первых двух, но для «Погашение долга» бессмысленно — зачем видеть тех,
кто ничего не должен.

Исправлено: при выборе «Погашение долга» список теперь показывает **только клиентов
с реальным открытым долгом** (переиспользую тот же список должников, что уже есть в блоке
«Должники» на этой же странице). Если должников вообще нет — показывается «Должников нет —
гасить нечего» вместо пустого списка.

## Финансы — переработана строка «Последние операции»

Было три отдельные колонки с равномерным разбросом (дата слева, описание по центру, сумма
справа) — из-за этого описание «болталось» посередине само по себе. Теперь дата, время и
описание объединены в один блок слева (если не помещается — обрезается многоточием, а не
переносится), сумма — отдельно справа. Ровно тот вид, что ты описал: «Дата время описание
… сумма».

## Клиенты — авитомагазин и заметки, Финансы/Касса — перевод системных текстов

### 1. Заметки о клиенте
Обнаружил, что таблица и бэкенд для заметок уже существовали (`client_notes`,
`GET`/`POST /clients/:id/notes`), но интерфейс их вообще не использовал. Добавил полноценный
раздел «📝 Заметки о клиенте» на карточке — поле ввода + список всех заметок с датой/временем
и именем сотрудника, который её оставил. Именно то самое «место, где можно оставлять заметки»,
чтобы не забывать, откуда клиент и что о нём важно помнить.

### 2. Поле «Авитомагазин»
Добавлено новое поле у клиента — название магазина на Avito. Доступно при создании клиента,
редактировании, и отображается на самой карточке (🛍️ рядом с источником привлечения).

### 3. Системные тексты операций теперь переводятся на китайский
В Финансах и Кассе тексты вида «Продажа (наличные)», «Погашение долга (курс 12.32)», «Сдача
обменнику» раньше писались бэкендом сразу на русском и не подхватывались переключением языка.
Сделал отдельную функцию-переводчик (`utils/translateNote.js`), которая распознаёт эти
фиксированные формулировки и переводит их, сохраняя вставленные числа/имена как есть.

## Клиенты: авитомагазин + заметки, Финансы: перевод системных текстов

### 1. Авитомагазин у клиента
Новое поле «Авитомагазин» — при создании клиента и в редактировании карточки. Показывается
на самой карточке (🛍️ рядом с источником привлечения).

### 2. Заметки о клиенте
Обнаружил, что таблица и бэкенд для заметок (`client_notes`) уже существовали в системе, но
интерфейс их не использовал вообще. Подключил: на карточке клиента теперь отдельный блок
«📝 Заметки о клиенте» — поле для быстрой записи (Enter или кнопка «Сохранить») + список всех
прошлых заметок с датой и именем того, кто написал. Именно то самое «место, где можно оставить
заметки», о котором ты говорил — откуда клиент, что за клиент и т.д.

### 3. Финансы — системные тексты операций теперь переводятся на китайский
Причина была в том, что тексты вроде «Продажа (наличные)», «Погашение долга (курс X)»,
«Сдача обменнику» пишутся бэкендом прямо в базу как русский текст — это данные, а не
интерфейс, поэтому обычный переключатель языка их не подхватывал.

Сделал отдельную умную функцию-переводчик (`utils/translateNote.js`), которая распознаёт эти
фиксированные формулировки (включая те, где встроен курс или имя обменника) и подставляет
перевод, не трогая сами цифры/имена. Подключил в **Финансы** («Последние операции») и **Кассу**
(показывают одни и те же данные).
